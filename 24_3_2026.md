## Synchronizace

Týká se procesů nebo vláken, která **sdílí paměť**.

Musí být splněno:
- **Konkurentní systém** s **preemptivním multitaskingem**
- Na úrovni **userspace**

Týká se **kritického kódu** → vede ke **kolizím** a kvůli špatné synchronizaci můžeme dostat **špatné výsledky**.

---

### Klasický příklad: Producent & Konzument

- **Producent** – zapisuje/přepisuje sdílenou paměť
- **Konzument** – očekává, že přečte celou (platnou) paměť

```
PRODUCENT → [sdílená paměť] → KONZUMENT
```

---

### Problémy se synchronizací

1. **Konzument běží první** – přečte nezapsanou (neplatnou) paměť
   → Producent musí běžet jako první (časová závislost)

2. **Producent zapíše 2×, konzument přečte jen 1×**
   → Ztráta dat (např. chybějící měření ze senzoru)

3. **Producent zapíše 1×, konzument přečte N×**
   → Duplicitní údaje (stejná data zpracována vícekrát)

---

### Vzájemné vyloučení (Mutual Exclusion) ⚠️ DŮLEŽITÝ TERMÍN

> **V jednom okamžiku může ke sdílené paměti přistupovat pouze jeden proces.**

- Procesy/vlákna nemohou přistupovat ke sdílené paměti současně
- Složitější scénáře: **více producentů a více konzumentů** → vzájemné vyloučení stále platí

```
Proces A  →|  kritická sekce  |→ ...
Proces B     (čeká, nemůže vstoupit současně)
```

> ⚠️ Vzájemné vyloučení **není největší cíl** synchronizace – mnoho lidí si to mylně myslí.
> Hlavní cíl je **správné pořadí**: producent musí běžet jako první.

---

### Co dál – jak synchronizace dosáhnout?

Probrali jsme problém producenta a konzumenta. Nyní se zaměříme na **mechanismy**, jak správné synchronizace dosáhnout.

---

### Synchronizační nástroje / Synchronizační primitiva

- Existují jako **SW i HW** řešení
- Slouží k zajištění vzájemného vyloučení nebo správného pořadí
- Dosáhneme toho právě pomocí těchto primitiv

---

### Binární semafor

> Nejstarší synchronizační primitivum – vzniklo na konci **50. let / začátku 60. let**.

- **Dvoustavový** – pouze dvě hodnoty:
  - 🔴 červená (zamčeno / obsazeno)
  - 🟢 zelená (volno / přístup povolen)
- ≠ semafor na křižovatce (jiný koncept)

#### Příklad: Vlaky a koleje

Klasická analogie pro vzájemné vyloučení:

```
        čeká                   jede
         |                      |
[🚂]====🔴====+================+====🟢====[🚂]>>
              |  jednokolejný  |
              |     úsek       |
              | (kritická sekce)|
              +================+
```

- Jednokolejný úsek = **kritická sekce** (sdílený zdroj)
- Semafor zajistí, že úsekem jede vždy **jen jeden vlak**
- Druhý vlak čeká, dokud úsek není volný

→ Přesná analogie vzájemného vyloučení v OS
→ Semafor na křižovatce je daleko komplexnější (chodci, více směrů...) – tato jednokolejná trať je ideální zjednodušení

---

#### Dvě operace nad semaforem

Různá jména dle literatury, Fišer používá:

| Operace | Alternativní názvy | Popis |
|---------|-------------------|-------|
| **WAIT** | acquire, down, P | Vstup do kritické sekce |
| **SIGNAL** | release, up, V | Výstup z kritické sekce |

#### WAIT – příjezd k semaforu

```
           🔴 ČERVENÁ → zastav a čekej na změnu stavu
vlak →→→ [?]
           🟢 ZELENÁ  → projeď a nastav semafor na 🔴
```

#### SIGNAL – odjezd od semaforu

```
vlak opouští úsek → nastav semafor na 🟢
```

#### Implementace WAIT v pseudokódu – naivní (ŠPATNĚ ❌)

```python
def wait(semafor):
    if semafor == RED:
        block(change)    # zastav a čekej na změnu stavu
    if semafor == GREEN:
        semafor = RED    # nastav červenou a vstup do kritické sekce
```

> ⚠️ **Toto nefunguje správně!**
> Mohlo by čekat více lokomotiv – po změně na 🟢 by **všechny** viděly zelenou a projely zároveň.
> Pokud bychom u `if` zůstali, musela by být implementována **čekací fronta**.

#### Implementace WAIT v pseudokódu – správně ✅

```python
def wait(semafor):
    while semafor == RED:  # po probuzení znovu zkontroluj stav
        block(change)      # zastav a čekej na změnu stavu
    semafor = RED          # nastav červenou a vstup do kritické sekce
```

> S `while` si každý probuzený proces sám znovu ověří stav → jen jeden projde.

#### Implementace v praxi

- V **userspace** je preemptivní multitasking složitý → správná implementace semaforu je náročná
- Proto se semafor implementuje v **jádře OS** s kooperativním přístupem
- S **jedním procesorem** to v jádře jde relativně snadno

#### Vícejádrové procesory – problém

- Více jader (např. 96jádrový procesor) → jádra spolu musí komunikovat a synchronizovat se navzájem
- **Zamykání sběrnice** = ostatní nesmí přistupovat ke sběrnici, i když přistupují k něčemu jinému → **zpomaluje celý systém**
- Řešení: **speciální HW podpora**
  - Zámek se aktivuje pouze tehdy, když **další procesor chce provést tu samou operaci**
  - Jinak ostatní jádra pokračují bez čekání
  - Operace musí být **atomická a rychlá**
  - Na základě této HW podpory jsme schopni implementovat semafor správně

> ⚠️ **DŮLEŽITÉ PRAVIDLO:**
> **Nikdy nepsat semafor běžnými operacemi!**
> Vždy použít buď:
> - **Jádro OS** (které poskytuje synchronizační primitiva) – např. mutexy, semafory
> - Pokud OS primitiva **neposkytuje** → použít **atomické instrukce** (HW podpora)

#### SIGNAL – implementace

- Jednoduchá operace – pouze nastav semafor na 🟢 zelená
- Žádná podmínka, žádné čekání

```python
def signal(semafor):
    semafor = GREEN    # nastav zelenou – kritická sekce je volná
```

#### Celý průběh

```
1. Vlak A přijede:  WAIT  → 🟢 zelená → vjede, nastaví 🔴
2. Vlak B přijede:  WAIT  → 🔴 červená → čeká
3. Vlak A odjíždí:  SIGNAL → nastaví 🟢
4. Vlak B pokračuje: WAIT  → 🟢 zelená → vjede, nastaví 🔴
```

### Použití semaforu – Producent & Konzument

> Semafory je lepší vytvářet v **zeleném stavu** (ne červeném).

#### Pouze vzájemné vyloučení (1 semafor) – nestačí ⚠️

```python
# jeden sdílený semafor (začíná GREEN)
while True:
    wait(s)       # zamkni
    write(data)   # zapiš do sdílené paměti
    signal(s)     # odemkni

# konzument
while True:
    wait(s)       # zamkni
    read(data)    # čti ze sdílené paměti
    signal(s)     # odemkni
```

> Problém: producent může zapsat 2× a konzument se nemusí dostat ke čtení → **pouze vzájemné vyloučení, ale ne časová synchronizace**.

#### Časová synchronizace (2 semafory) – správně ✅

Každý čeká na **svém** semaforu, ale signalizuje ten **opačný**.

```
Počáteční stav:
  sem_producent = 🟢  (producent může hned psát)
  sem_konzument = 🔴  (konzument čeká, až producent zapíše)
```

```python
# producent
while True:
    wait(sem_producent)   # čekej než smím psát
    write(data)           # zapiš
    signal(sem_konzument) # dej vědět konzumentovi

# konzument
while True:
    wait(sem_konzument)   # čekej než smím číst
    read(data)            # přečti
    signal(sem_producent) # dej vědět producentovi
```

```
  sem_producent=🟢          sem_konzument=🔴
       |                          |
       ▼                          ▼
  [PRODUCENT]                [KONZUMENT]
  wait(sem_P) ──🟢──►  write(data)
                              │
                         signal(sem_K) ──► sem_konzument=🟢
                                                │
                                           [KONZUMENT]
                                           wait(sem_K) ──🟢──► read(data)
                                                                    │
                                                               signal(sem_P) ──► sem_producent=🟢
                                                                                      │
                                                                                 (opakuje se)
```

#### Symetrie kódu – záleží jen na pohledu

Kód producenta a konzumenta vypadá **symetricky** – bez kontextu nevíme, kdo je kdo.
Záleží pouze na tom, **jak situaci chápeme**:

| Pohled | Producent | Konzument |
|--------|-----------|-----------|
| Klasický | zapisuje data | čte/maže data |
| Duální | produkuje plný prostor | produkuje **volný** prostor |

> Konzument tím, že přečte/smaže data, **produkuje volný prostor** pro producenta.
> Producent tím, že zapíše, **produkuje plný prostor** pro konzumenta.
>
> Kód je tedy symetrický – jde jen o to, **jak ho pojmenujeme a jak mu rozumíme**.

#### Příklad: Hostinský a hosté 🍺

- **Hostinský** plní půllitry pivem → produkuje plné půllitry
- **Hosté** pijí a vracejí prázdné půllitry → produkují prázdné půllitry

Nebo duálně:
- Hosté produkují **prázdné půllitry** a hostinský je konzumuje (tím že je plní, přestávají být prázdné)

> V realitě to ale **symetrické není** – intuitivně víme, kdo je producent a kdo konzument.
> Symetrie je jen matematická vlastnost kódu, ne reálného problému.

### Nevýhody binárního semaforu – proč se dnes nepoužívá

#### Deadlock (uváznutí)

> Pokud zavoláme `wait()` **dvakrát** nad stejným semaforem → **DEADLOCK**

- Proces čeká sám na sebe, aby semafor uvolnil
- Nekonečná závislost → program **zmrzne**
- Deadlock může vzniknout vždy, i omylem (zapomeneme zavolat `signal`, voláme `wait` 2×, ...)
- Poznáme ho tak, že **program přestane reagovat**

```
wait(s)   ← zamkne
  ...
wait(s)   ← čeká na uvolnění... ale kdo ho uvolní? Nikdo → DEADLOCK ∞
```

### Nadstavba nad semafory – Monitor / synchronized

Nad semafory existují **komplexnější prostředky** – např. **synchronizační monitor**.

#### Java – klíčové slovo `synchronized`

```java
synchronized void metoda() {
    // vzájemné vyloučení zajištěno automaticky
    // při vstupu se zamkne, při opuštění se uvolní
}
```

- Při volání `synchronized` metody se **automaticky zamkne** objekt
- Při opuštění se **automaticky uvolní** → nelze zapomenout zavolat `signal`
- Proto je to **bezpečnější než semafor**

#### Ale pozor – deadlock i u monitoru! ⚠️

```java
synchronized void metodaA() {
    metodaB();   // volá jinou synchronized metodu stejného objektu
}

synchronized void metodaB() {
    // pokus o zamknutí stejného zámku → DEADLOCK
}
```

- Pokud uvnitř `synchronized` metody zavoláme **jinou `synchronized` metodu** se stejným zámkem → **deadlock**
- Nejčastější příčina: **vnořená volání** metod stejného objektu
- Buď složitá logika, nebo to prostě **nebude fungovat**

#### Další nevýhoda – víceúrovňové systémy

- Více producentů a konzumentů (např. 8 producentů, 8 konzumentů)
- Musíme správně volat `wait`/`signal` pro každého → **velmi náchylné na chyby**
- Pro složitější úlohy jsou semafory **těžkopádné a nepraktické**

### Moderní synchronizační primitiva

Místo binárního semaforu se dnes používají dvě specializovaná primitiva:

| Primitivum | Účel |
|------------|------|
| **Mutex** | Vzájemné vyloučení |
| **Synchronizační událost (Event)** | Časová synchronizace (správné pořadí) |

- **Mutex** – zajistí, že ke kritické sekci přistupuje vždy jen jeden proces/vlákno
- **Event** – zajistí, že jeden proces počká, dokud druhý nedokončí svou práci (producent → konzument)
